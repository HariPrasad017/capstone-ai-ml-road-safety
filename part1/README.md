# Part 1 — Data Preparation & Exploration
## Road Accident Severity Analysis (India)

## What is this project about?

Every year, thousands of road accidents happen across India, and some are far
more serious than others. This project looks at a real dataset of 3,000 road
accidents recorded across Indian states and tries to answer a simple question:

> **"What factors make an accident more likely to be serious or fatal, instead
> of minor?"**

For example — does drinking and driving really lead to worse accidents? Does
poor lighting at night make things more dangerous? Which states report the
most accidents? This project cleans the raw data, explores it with charts,
and pulls out clear answers to questions like these — the kind of insight
that could genuinely help road safety planning.

This is **Part 1 of a 4-part project**. Part 1 focuses only on preparing and
understanding the data. Later parts (not in this folder) build a prediction
model and an AI system on top of this work.

## Where did the data come from?

- **Source:** [Road Accident Severity in India — Kaggle](https://www.kaggle.com/datasets/s3programmer/road-accident-severity-in-india)
- **Size:** 3,000 accident records, 22 pieces of information (columns) per record
- **What each row represents:** One road accident, with details like which
  state/city it happened in, the weather, road type, whether alcohol was
  involved, how many people were hurt, and how severe the accident was
  classified as (Minor / Serious / Fatal)

## What did we have to fix in the raw data, and why?

Real-world data is never perfectly clean. Here's what we found and how we
handled it — explained in plain terms:

1. **"Traffic Control Presence" was missing for 716 accidents (about 24%).**
   Rather than guessing randomly, we assumed a missing entry most likely
   means *no traffic signal, sign, or checkpost was present* at that
   location — since if something wasn't there, it wouldn't have a value.
   So we filled these blanks with `"None"` instead of throwing away nearly
   a quarter of our data.

2. **"Driver License Status" was missing for 975 accidents (about 33%).**
   We couldn't safely guess this one (a missing value here doesn't obviously
   mean "no license"), so we labelled these as `"Unknown"` — this keeps the
   data honest about what we don't know, while still keeping the row usable
   for every other analysis.

3. **Checked for duplicate accident records** — none were found, so no rows
   needed to be removed for this reason.

4. **Checked that numbers were stored as numbers** (e.g., number of
   casualties, driver age, speed limit) and that categories were stored as
   text (e.g., state name, weather) — everything was already correctly
   formatted.

**Why does this matter?** If we had simply deleted every row with a missing
value, we would have thrown away over 1,500 of our 3,000 records — more than
half the data — for no good reason. Filling in sensible values instead lets
us keep almost all the real information we have.

## Limitations — what this data *can't* tell us

Being upfront about limitations is part of doing honest analysis:

- The data doesn't include exact GPS location or road names, so we can only
  analyze at the state/city level, not pinpoint dangerous stretches of road.
- Some fields (like "cause of accident") are recorded by whoever filed the
  report, so there's a chance of human judgment affecting the labels.
- This dataset is a sample, not every accident that happened in India — so
  findings describe patterns *in this dataset*, not a guaranteed nationwide
  statistic.
